# Fixed WebRTC Audio-Only Connection Issue

I have implemented a series of fixes to resolve the issue where audio-only calls were failing to establish a WebRTC connection while video calls worked correctly.

## Changes Made

### 1. Android: Explicit Audio Transceiver & SDP M-Lines
- **`WebRtcManager.kt`**: Added logic to explicitly add an **Audio Transceiver** (`MediaStreamTrack.MediaType.MEDIA_TYPE_AUDIO`) with `SEND_RECV` direction when a call is audio-only. This ensures that the generated SDP always contains the necessary audio M-lines even before any local tracks are fully processed.

### 2. Android: Explicit Media Constraints
- **`CallRoomFragment.kt`**: Updated the `createOffer` call to use explicit `MediaConstraints`.
    - `OfferToReceiveAudio` is now hardcoded to `true`.
    - `OfferToReceiveVideo` is set to `true` only for video calls and `false` for audio calls.
    - This ensures the browser understands exactly what the Android client expects to receive.

### 3. Android: Signaling Fallback Trigger
- **`CallRoomFragment.kt`**: Implemented a **1.5-second fallback delay** in `activateCallSession()`. If the web client doesn't send the `web-ready` signal within this window, the Android app will automatically trigger `onWebReady()` and send the SDP Offer. This prevents the call from hanging if the signaling event is lost or delayed.

### 4. Web: Reliable Ready Signal
- **`index.html`**: Moved the `web-ready` broadcast signal to trigger **immediately after** the `RTCPeerConnection` is initialized and local tracks are added. This bypasses potential delays or missed callbacks from the Supabase channel subscription.

### 5. Robust State Management
- **`CallRoomFragment.kt`**: Added an `isOfferSent` atomic flag to prevent duplicate SDP Offers from being generated by both the web signal and the fallback timer.
